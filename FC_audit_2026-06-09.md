# Betaflight F405 FC — Project Audit (2026-06-09)

**Scope:** Betaflight/STM32 path only. Audited against `drone_fc_project_context.md` (v1 on disk), prior-session corrections (LQFP-64, TLV733P, PB8 LED, SWD header), and your answers today: ESC = Flycolor Raptor BLS-04, no video system (bench/resume build).

**Verdict: Proceed with changes.** The architecture is sound and the MCU choice is right. But the schematic must not start from the v1 doc on disk — it contains two components that would brick the board (missing VCAP caps, missing BOOT1 strap), an unverified ESC pinout sourced from the wrong manual, and a telemetry feature your ESC firmware doesn't have.

---

## Critical

**C1. The ESC manual in the project is for a different ESC.**
The uploaded PDF (Flycolor doc 251400-1168-TX) is the **Trinx G20 4-in-1**: BLHeli_32, STM32G0, 20×20mm mount, pinout order TX/CRT/S1/S2/S3/S4/NC/VBAT/GND/GND. Your ESC is the **Raptor BLS-04** (tower product: BLHeli_S, 30.5×30.5mm, 60–65A/70A, 3–6S). No public text-searchable manual for the BLS-04 harness exists; the pinout table in the context doc matches the Trinx manual read **end-to-end reversed**, with pin 10 labeled NC where the Trinx puts telemetry TX. Provenance unknown = treat as unverified.
*Fix:* Before any schematic work, verify the physical unit: (1) photograph the ESC silkscreen at the 10-pin SH1.0 socket; (2) with battery disconnected, continuity-test each socket pin to the XT60 pads (finds VBAT and the two GNDs) and to each motor MOSFET cluster (finds S1–S4); (3) verify at the **FC end of the supplied cable** — a 1:1 SH1.0 cable mirrors pin order between the two boards. Lock the verified table into the v3 doc. Rationale (per project rule): a one-position error puts up to 25.2V on a 3.3V GPIO.

**C2. VCAP capacitors are missing from the design.**
The STM32F405's internal 1.2V core regulator requires a 2.2µF capacitor on **each** VCAP pin (VCAP_1 = pin 31, VCAP_2 = pin 47 on LQFP-64). The component list has decoupling for VDD/VDDA only — no VCAP caps anywhere. Without them the core supply is unstable and the MCU will not run reliably, if at all.
*Fix:* Add 2×2.2µF X5R/X7R (≥6.3V, 0603) placed immediately at the VCAP pins. Ref: STM32F405 datasheet (DS8626) §"Voltage regulator" and ST AN4488 hardware getting-started.

**C3. No BOOT1 (PB2) strap — DFU flashing will be unreliable.**
The plan has BOOT0 (10k pull-down + button) but never mentions PB2/BOOT1. Entering the ROM bootloader (your no-debugger flash path: BOOT0 button + USB DFU) requires BOOT0=1 **and** BOOT1=0. Floating PB2 means random boot target (system memory vs SRAM). Ref: ST AN2606 boot mode table.
*Fix:* 10k pull-down on PB2 (or tie direct to GND if you'll never use it as GPIO).

**C4. The context doc on disk is stale and would propagate fixed bugs.**
`drone_fc_project_context.md` still says LQFP-100 (the RGT6 is LQFP-64), single AP2112K (no TLV733P IMU LDO), LED on PC13 (3mA power-switch-domain pin; remapped to PB8 last session), and no SWD header. The corrected v2 was saved to a temporary session folder that no longer exists.
*Fix:* Regenerate a v3 context doc incorporating v2 fixes + this audit before opening KiCad. I can produce it on request.

## Significant

**S1. ESC telemetry header is a dead feature — BLHeli_S has no serial telemetry.**
The plan dedicates UART3 + a 3-pin header to "ESC telemetry (optional)". The BLS-04 runs BLHeli_S, which has no UART telemetry output (that's BLHeli_32/AM32/KISS territory). The wire would carry nothing.
*Fix:* Drop the dedicated telemetry header. Get RPM data the modern way instead: Flycolor's BLHeli_S towers use the **EFM8BB21F16** ESC MCU (per the Raptor S-Tower manual), which is supported by **Bluejay** — flash via ESC-Configurator to get bidirectional DSHOT → RPM filtering in Betaflight. Verify the BLS-04's exact layout in ESC-Configurator before flashing. Keep UART3 routed to pads as a spare (GPS/MSP later).

**S2. No 5V input or distribution on the board.**
The power chain names the external Matek MBEC6S, but the connector list has no 5V input pads, the receiver header has no 5V pin (an ELRS RX needs 5V, ~50–100mA), and there's no isolation between USB VBUS and the MBEC's 5V — on the bench with USB + battery connected, two regulated 5V sources fight each other.
*Fix:* Add 5V/GND input solder pads; add 5V to the receiver header; OR the two sources with a Schottky (e.g., SS34/B5819W) from VBUS into the 5V rail so USB can power MCU+RX for configuration, and the MBEC wins in flight. Onboard buck is a fine v2 upgrade but unnecessary now — the MBEC6S is owned and within spec (6–30V in, 1.5A).

**S3. Buzzer type: plan specifies a passive buzzer, Betaflight drives active ones.**
Betaflight's beeper is an on/off GPIO, not a tone generator — a passive transducer will click, not beep.
*Fix:* Spec a 5V **active** magnetic buzzer; keep the 2N7002 low-side switch + 1N4148 flyback. Set beeper polarity in the target config at bring-up.

**S4. Motor numbering must follow the verified harness, and timer/DMA config shouldn't be invented from scratch.**
Your four pins (PA2, PA3, PB0, PB1) are exactly the OMNIBUSF4 motor set — a decade-proven F405 target — but its mapping is Motor1=PB0, Motor2=PB1, Motor3=PA3, Motor4=PA2, the reverse of your table, and the true S1–S4 order is unknown until C1 is done.
*Fix:* Copy the OMNIBUSF4 `TIMER_PIN_MAP` rows for these pins verbatim (known-good DSHOT DMA assignments), then set `MOTORx_PIN` to match the verified harness. Reordering happens in config, not copper.

**S5. Current-sense scale is unknown.**
No datasheet states the BLS-04's CRT output V/A. The 1kΩ series resistor + Zener clamp plan is right (keep the Zener — 3.3V on a 65A sensor line is cheap insurance), but the Betaflight `current_meter_scale` must come from measurement.
*Fix:* Bench-calibrate post-build: measure CRT voltage at known currents (current clamp or charger-logged mAh comparison), set scale/offset in CLI.

## Minor

**M1. 22Ω USB series resistors are unnecessary on F405.** AN4879: the FS transceiver manages line impedance internally; no external series resistors needed. Harmless if kept; deleting saves two parts. Optional: USBLC6-2SC6 ESD array on D+/D−.
**M2. VBUS sense on PC0 is optional.** Betaflight supports it (`USB_DETECT_PIN`, cf. OMNIBUSF4 on PC5) but most F405 boards omit it. Keep or drop freely.
**M3. Crystal load caps: confirm against the chosen crystal's CL.** 2×22pF suits CL≈12–14pF with ~2–3pF stray. If you buy a CL=18pF part you'd need ~30pF. Check before ordering.
**M4. Add a 10k pull-up on IMU CS** (PC4→3.3V) so the ICM-42688-P can't see a floating CS during boot/DFU.
**M5. Spare pads cost nothing:** USART6 (PC6/PC7) and I2C1 (PB6/PB7, future baro/mag) broken out to pads keeps a 5th UART and sensor expansion open on a bench board that may grow.
**M6. MCU VBAT pin → 3.3V + 100nF** (not in the component list; required even if RTC unused).
**M7. Mounting confirmed good:** BLS-04 tower is 30.5×30.5 M3 — your hole pattern matches. (Had the ESC been the Trinx G20 of the uploaded manual, it would not have: 20×20.)

## The MCU question: keep the STM32F405RGT6

Checked alternatives (June 2026 data):

| MCU | LCSC price/stock | Betaflight status | Assessment |
|---|---|---|---|
| **STM32F405RGT6** | ~$2.12–2.61, deep stock, JLCPCB-assemblable (C15742) | Target platform present in master; wiki: "most common option" | **Keep** |
| AT32F435RGT7 | ~$2.70, only 362 in stock | Official since 4.5; pin-compatible with F405 | Fallback if F405 dries up; same footprint |
| STM32F722 | similar price, spottier stock | Supported | No meaningful gain for this project |
| STM32H743 | ~3–5× price, LQFP-100 | Supported | Overkill; harder layout, no resume gain |
| STM32G473 | cheap | "Not as developed and established" (BF wiki) | Weak community base |

F405 deprecation risk is low: Betaflight moved to year-based releases (4.5.x → 2025.12 → 2026.6) and F405 remains a first-class platform with the largest reference-design base — which is the point of this project. The pin-compatible AT32F435 means your exact footprint has a second source. No chip change recommended.

## Completion roadmap

1. **Verify ESC harness** (C1 procedure) → lock pinout table.
2. **Regenerate context v3** folding in v2 fixes + C2/C3/S1–S5/M-items.
3. **Schematic in KiCad** in the doc's build order, with additions: VCAP caps, PB2 strap, 5V pads + ORing diode, RX-header 5V, active buzzer, spare-UART/I2C pads. Run ERC.
4. **Layout:** IMU dead-center on solid L2 ground, TLV733P close to IMU, crystal short loop, USB pair length-matched, VCAP/decoupling at pins, motor signals away from IMU. DRC against JLCPCB 4-layer rules.
5. **Order** (JLCPCB economic PCBA; check PCBWay only if F405/IMU reels go out of stock).
6. **Bring-up, props off, ESC disconnected first:** smoke-test 5V/3.3V rails → SWD connect (ST-Link, `STM32_Programmer` or probe via DFU: BOOT0 held, USB) → flash → Configurator sensor check → blackbox flash test → only then plug verified ESC harness, motors spin-tested without props.
7. **Custom target:** clone `betaflight/betaflight` + `betaflight/config`; create `configs/ETHANF405/config.h`; `make configs && make ETHANF405`. Skeleton:

```c
#define FC_TARGET_MCU     STM32F405
#define BOARD_NAME        ETHANF405
#define MANUFACTURER_ID   CUST   // local-build only; rules apply only for official submission

#define USE_GYRO
#define USE_GYRO_SPI_ICM42688P
#define USE_ACC
#define USE_ACC_SPI_ICM42688P

#define GYRO_1_SPI_INSTANCE SPI1
#define GYRO_1_CS_PIN       PC4
#define GYRO_1_EXTI_PIN     PC3
#define FLASH_SPI_INSTANCE  SPI2
#define FLASH_CS_PIN        PB12

// Motor pins: set 1..4 to the bench-verified S1..S4 after C1.
// TIMER_PIN_MAP rows for PB0/PB1/PA3/PA2: copy from OMNIBUSF4 config.h (known-good DSHOT DMA).

#define ADC_CURR_PIN        PC2
```

8. **Calibrate** vbat scale (110 nominal for 100k/10k) and current scale (S5). Flash Bluejay on the ESC if RPM filtering is wanted.
